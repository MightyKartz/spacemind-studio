# 商业空间 AI 头脑风暴工具开发方案

日期：2026-05-12  
项目定位：面向商业空间方案阶段的 AI test-fit 与可编辑图纸生成工具

## 1. 一句话定位

把原始商业空间图纸转成结构化空间数据，让 AI 参考历史案例库，在建筑约束内快速生成多套可编辑功能落位方案，帮助设计师、品牌方、招商和租赁团队在早期方案阶段更快完成头脑风暴、比选和汇报。

推荐对外表达：

> 上传铺位图，30 秒生成多套商业空间功能落位方案，并导出可编辑汇报图。

## 2. 为什么值得做

当前商业空间早期方案有几个明确痛点：

- 原始平面图到功能落位图通常需要设计师手动拆图、描边、摆家具、上色、写说明。
- 方案阶段经常要快速出多版，但人力有限，导致头脑风暴不足。
- 过往优秀案例沉淀在 Revit、CAD、PDF、PNG、PPT 里，很难被复用。
- 品牌方、招商、租赁团队想快速判断空间潜力，但不一定会操作 CAD 或 Revit。
- 通用 AI 作图容易生成漂亮但不可落地的图，无法保证墙体、门洞、柱子、通道、后场等约束。

本项目的核心价值不是替代最终施工图，而是把“早期方案探索”从数小时压缩到数分钟。

## 3. 目标用户

第一优先级：

- 商业空间设计公司
- 室内设计工作室
- 连锁品牌门店设计部
- 咖啡、餐饮、零售、展厅品牌拓展团队
- 商场招商主管、资产运营团队
- 办公空间、联合办公、写字楼租赁服务商

第二优先级：

- 装修施工公司
- 家具、软装、道具供应商
- 设计院方案团队
- 建筑与室内设计教学机构

早期不建议优先做：

- 普通 C 端家装用户，竞争过于拥挤。
- 施工图深度交付，责任和精度要求过高。
- 大型城市规划和复杂建筑体量生成，容易撞上成熟 AEC 平台。

## 4. 产品边界

第一版应该做：

- 上传原始图纸
- 识别或手动确认边界、入口、柱子、后场等约束
- 输入项目需求
- 从历史案例库检索相似案例
- 生成 3-8 个可编辑方案
- 提供方案评分和设计说明
- 在 2D 编辑器中继续调整
- 导出 PNG、PDF、SVG、JSON

第一版不应该承诺：

- 一键生成最终施工图
- 完全自动理解所有 Revit 族和复杂标注
- 从 PNG 100% 准确还原建筑语义
- 自动承担规范、消防、审图责任

## 5. 典型用户流程

```text
用户上传 PNG/PDF/DXF/RVT
  ↓
系统生成底图和空间语义 JSON
  ↓
用户确认边界、入口、人流、固定区域
  ↓
用户输入经营目标和功能需求
  ↓
系统检索相似历史案例
  ↓
AI 生成多套设计策略
  ↓
几何与约束引擎生成可编辑布局
  ↓
系统自动评分、解释、渲染方案卡片
  ↓
用户选择方案并对话修改
  ↓
导出汇报图、结构化 JSON、PDF/SVG/DXF
```

## 6. 交互设计建议

建议做成 Web App，核心界面为：

```text
左侧：2D 图纸画布
右侧：AI 侧边栏
底部或右侧：方案卡片列表
顶部：项目、导入、生成、导出、图层、版本
```

关键交互：

- 上传图纸后，系统自动框出可设计区域。
- 用户可以点击确认或手动修正边界。
- AI 侧边栏以问答方式收集需求。
- 生成后展示多个方案卡片，每个卡片包含预览、策略说明、评分。
- 用户可以说“参考第 3 个方案，但减少办公区，增加舒适软座”。
- 方案中的色块、家具、文字、箭头都必须可编辑。
- 用户最终选择的方案和修改记录进入案例库。

## 7. 用户需要输入的内容

必填：

- 项目类型：咖啡、零售、办公、餐饮、展厅、快闪等。
- 经营目标：展示更强、座位更多、社交属性更强、动线更顺、成本更低等。
- 入口位置和人流比例。
- 固定区域：后场、设备间、不可动墙、柱子、消防或设备限制。
- 需要的功能区：吧台、卡座、散座、办公区、储藏、展示区、洽谈区等。

可选：

- 预算等级
- 品牌调性
- 风格偏好
- 参考案例
- 最少座位数
- 通道最小宽度
- 是否偏汇报表达或偏施工落地

## 8. 支持导入的内容

推荐优先级：

1. PNG/PDF：最快落地，作为底图和手动校准基础。
2. DXF/SVG：保留矢量线条，识别边界和墙体更准确。
3. RVT：通过本地 Revit 插件或 Autodesk APS 后台转换成空间 JSON。
4. 历史案例：导入已有成品图、Revit、CAD、PDF、JSON，并补充标签。

早期建议先支持 PNG/PDF + 人工确认边界，再逐步接入 DXF/SVG/RVT。

## 9. 最终输出

用户可获得：

- 方案预览图：类似功能分区汇报图。
- 可编辑图纸：前端画布 JSON。
- 空间语义 JSON：用于案例库和后续 AI 生成。
- PNG/PDF：用于汇报。
- SVG/DXF：用于继续进入设计工作流。
- 方案说明：每个功能区为什么这样放。
- 评分报告：动线、容量、展示面、成本、施工复杂度。
- 修改记录：用于后续训练和案例沉淀。

## 11. 为什么 LLM 不能直接画图

LLM 适合：

- 理解项目需求
- 总结场地条件
- 检索和解释相似案例
- 生成设计策略
- 生成方案说明
- 根据对话修改方案意图

LLM 不适合单独负责：

- 精确坐标
- 家具碰撞
- 通道宽度
- 门扇开启范围
- 柱子避让
- 建筑边界裁剪

正确做法：

```text
LLM 输出布局意图和参数
几何引擎生成坐标
约束引擎做校验
前端渲染成可编辑图纸
```

## 13. Revit/RVT 转 JSON 路线

推荐三阶段：

### 阶段 1：本地 Revit 插件导出

- 用 C# Revit Add-in 读取 RVT。
- 导出墙、门、柱、房间、家具、文字、填充区域、视图信息。
- 保存为项目标准 JSON。
- 适合内部批量处理历史案例。

### 阶段 2：网页上传 + 后台队列

- 用户在网页上传 RVT。
- 后端保存到对象存储。
- 后台任务调用转换服务。
- 转换完成后写入数据库。

### 阶段 3：Autodesk APS 云端转换

- 使用 Autodesk Platform Services Automation API for Revit。
- 把 Revit Add-in 打包成 AppBundle。
- 配置 Activity。
- 每次上传 RVT 创建 WorkItem。
- 云端 RevitCoreConsole 打开 RVT 并运行插件。
- 输出 JSON，后端下载并入库。

注意：APS 需要付费，适合产品化后使用。早期可先用本地插件处理历史数据。

## 14. 案例库如何越用越准

最有价值的数据不是成品图，而是完整链路：

```text
原始图纸
→ 场地条件
→ 用户需求
→ AI 生成方案
→ 用户选择
→ 用户修改 diff
→ 最终方案
→ 方案评分和复盘
```

需要记录：

- 用户选择了哪个方案
- 用户删除了哪些功能区
- 用户移动了哪些家具
- 用户让 AI 修改了什么
- 最终导出的版本
- 客户或团队是否认可

这些数据后期可用于：

- 相似案例检索
- 方案排序模型
- 用户偏好模型
- 领域 LLM 微调
- 专用布局生成模型

## 15. MVP 开发计划

### MVP 目标

让用户上传一张原始平面图，在 30 秒内得到 3-6 个可编辑商业功能落位方案，并导出汇报图。

### MVP 范围

- Web 端项目创建
- PNG/PDF 上传
- 底图锁定
- 比例尺校准
- 手动绘制可设计边界
- 标记入口、人流、柱子、后场
- 输入项目 brief
- 简易案例库检索
- AI 生成方案策略
- 几何引擎生成分区色块
- 基础家具符号库
- 方案卡片
- 对话式局部修改
- PNG/PDF/JSON 导出

### 暂缓功能

- 完整 RVT 云端转换
- 自动施工图生成
- 3D 渲染
- 多人协作
- 精细算量报价
- 企业权限和审计

## 17. 方案评分维度

建议每个方案自动给出：

- 动线效率
- 容量
- 展示面
- 入口吸引力
- 功能完整度
- 成本复杂度
- 施工可落地性
- 品牌表达匹配度

评分不是绝对结论，而是帮助用户快速比较。

## 20. 风险与应对

风险 1：图纸识别不准  
应对：MVP 允许用户手动确认边界、入口和固定区域。

风险 2：AI 生成漂亮但不可用  
应对：LLM 只生成策略，几何和约束引擎负责坐标和校验。

风险 3：历史案例质量不统一  
应对：建立案例入库规范和人工校正流程。

风险 4：Revit 转换周期长  
应对：早期先用 PNG/PDF + 手动确认，RVT 转换作为第二阶段。

风险 5：用户不信任 AI 方案  
应对：输出方案来源、参考案例、评分和可解释说明。

## 21. 关键成功指标

产品效率指标：

- 从上传图纸到生成首批方案的时间小于 60 秒。
- 生成 6 个方案的平均时间小于 3 分钟。
- 用户能在 15 分钟内导出第一版汇报图。

质量指标：

- 方案不穿墙、不挡门、不压柱。
- 主通道宽度符合用户设置。
- 功能区能被用户理解并继续编辑。

商业指标：

- 用户生成后至少选择一个方案继续编辑。
- 用户愿意上传第二个项目。
- 团队愿意导入历史案例库。
- 最终方案被导出或用于汇报。

## 22. 下一步行动

建议先做 4 周 POC：

1. 建立图纸 JSON schema。
2. 做静态 Web 画布原型。
3. 支持导入 PNG 并锁定底图。
4. 支持手动绘制边界、入口、柱子、后场。
5. 准备 20-50 个历史案例，人工标注为结构化 JSON。
6. 接入 LLM，生成方案策略和分区 JSON。
7. 前端渲染 3-6 个方案卡片。
8. 支持导出 PNG 和 JSON。

验证标准：

- 真实设计师是否愿意用它做前期脑暴。
- 生成方案是否能激发讨论。
- 用户是否愿意继续编辑而不是直接放弃。
- 历史案例检索是否明显提升方案质量。
